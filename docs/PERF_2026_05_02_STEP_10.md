# Step 10 — vectored writev for HTTP/1 dispose hot path

**Branch**: `feat/multishot-alloc-cb`
**Date**: 2026-05-02
**Bench**: `wrk -t4 -c64 -d8s` over docker `benchnet`, 1 worker, 7 runs per cell.

## What landed

A new vectored fire-and-forget IO API in php-src + ext/async, plus a
threshold-driven hot-path branch in the HTTP/1 dispose: small responses
keep using the legacy concat formatter; responses ≥ `HTTP_WRITEV_THRESHOLD`
(1 KiB) skip the headers+body memcpy and submit the two zend_string
parts directly to the kernel via `writev`.

### Public API additions (php-src)

```c
typedef zend_async_io_req_t *(*zend_async_io_writev_t)(
    zend_async_io_t *io,
    zend_string * const *bufs, unsigned nbufs);

#define ZEND_ASYNC_IO_WRITEV(io, bufs, nbufs) ...
```

Each entry of `bufs` is an OWNED `zend_string` reference. The reactor
consumes one ref per entry on completion via `zend_string_release()` —
no separate `free_cb`/`ctx` machinery, no raw-buffer ownership tracking.
Interned strings (e.g. ad-hoc `"\r\n"` terminators) are free because
their `release` is a no-op.

### Server-side changes (true-async-server2)

- New `http_response_format_parts(obj, &headers, &body)` — emits the
  status line + Content-Length + headers as one `zend_string` and hands
  back a refcount-bumped reference to the body string. Saves the
  emalloc + memcpy of body bytes that the legacy `http_response_format`
  spends concatenating headers and body into a single buffer.
- New `http_connection_send_strv_owned(conn, bufs, nbufs)` — thin wrap
  around `ZEND_ASYNC_IO_WRITEV` for the plaintext path.
- `http_handler_coroutine_dispose` (plaintext branch) gates on
  `body_len < HTTP_WRITEV_THRESHOLD`: small → concat, large → writev.
  TLS path keeps the single-buffer formatter (the encrypt ring needs
  contiguous bytes, vectored would just force an extra copy).

## Why a threshold and not "always writev"

A first prototype unconditionally routed every response through writev.
On the minimal "ok" benchmark this was a **net loss** — the writev
submit overhead (pecalloc with variable size + slot loop +
io_pipe_writev_cb release loop, ≈150 ns total) exceeded the savings
from skipping a 2-byte memcpy (a few ns).

Perf snapshot, "ok" body (always-writev variant):
```
1.17%  _emalloc                         (was 1.20% on legacy concat path)
0.79%  _efree                           (was 0.68%)
0.63%  libuv_io_writev                  ← new
0.30%  io_pipe_writev_cb                ← new
0.27%  writev (libc syscall)            ← new
```

So writev paid +1.2% CPU in new symbols and only saved 0.04% in
`_emalloc`. Net negative for a 2-byte body.

## A/B by body size — where the crossover actually sits

7 wrk runs per cell. `tas-nowritev` is a build with
`HTTP_WRITEV_THRESHOLD = 256 MiB` (effectively never writev);
`tas-writev` had threshold 256 (always writev for body ≥ 256). Median
of 7 reported.

| body  | concat rps | writev rps | Δ rps  | concat p99 | writev p99 | Δ p99 |
|------:|-----------:|-----------:|-------:|-----------:|-----------:|------:|
| /4k   |  51 973    |  53 276    | +2.5%  | 2.93 ms    | 2.83 ms    | −3.4% |
| /16k  |  30 838    |  30 892    | +0.2%  | 3.45 ms    | 3.40 ms    | −1.4% |
| /64k  |  24 820    |  29 287    | **+18.0%** | 4.03 ms    | 3.84 ms    | −4.7% |
| /256k |  11 754    |  14 529    | **+23.6%** | 7.84 ms    | 7.08 ms    | −9.7% |

Reading:
- /4k–/16k: writev is wash on rps with a small p99 improvement (1–4%).
  Likely because at this size kernel-side syscall throughput dominates
  over the per-call CPU diff in either direction.
- /64k+: writev clearly wins. At 64 KiB, the avoided memcpy is ~6 µs of
  L1-bound work per response — that is finally large enough to dominate
  the writev submit overhead on the CPU side.

This is much higher than the back-of-envelope point-of-crossover I
estimated (≈500 B). The reason: small_allocator paths in Zend MM are
fast and amortise, and at /4k–/16k the throughput is already
syscall-bound rather than CPU-bound.

## Threshold choice — 1 KiB

Set to 1 KiB:

- Every "hello world" / JSON-stub response (say ≤ a few hundred bytes)
  stays on the legacy concat path → **no regression** on the
  microbench class that motivated step 4.1.
- 1 KiB cleanly excludes "/4k" and below from writev where the win is
  noisy/wash, but lets /16k+ benefit from the p99 improvement on the
  occasional large response.
- 64 KiB+ payloads see the full +18–24% rps win.

A more aggressive threshold (e.g. 256) would not regress small
responses on the bench but adds writev overhead in the wash zone for
no upside. A more conservative threshold (e.g. 16 KiB) would leave the
4–16 KiB p99 improvement on the table. 1 KiB is the simplest pick that
gives all the win and none of the risk.

## Sanity check at threshold = 1024

5 wrk runs per cell on the final landing.

| endpoint | rps median | p99 median | path taken |
|---|---:|---:|---|
| `/`     | 52 690 | 2.82 ms | concat (body=2, < 1024) |
| `/64k`  | 29 304 | 3.89 ms | writev |

`/` is in the same noise envelope as the step 4.1 baseline (~55k rps,
~2.8 ms p99). `/64k` is identical to the writev column of the A/B
matrix above.

## Reproducing

```bash
# Inside docker, with a server-variant.php that maps:
#   GET /     → 'ok'              (2 bytes, threshold→concat)
#   GET /4k   → str_repeat('x', 4096)
#   GET /16k  → str_repeat('x', 16384)
#   GET /64k  → str_repeat('x', 65536)
#   GET /256k → str_repeat('x', 262144)

docker run -d --rm --name tas-bench --network benchnet \
    -e CODEL_MS=0 -e TELEMETRY=0 tas-step10b \
    php -d extension=true_async_server.so /app/server-variant.php

docker run --rm --network benchnet williamyeh/wrk \
    -t4 -c64 -d8s --latency http://tas-bench:8080/64k
```

A/B builds (threshold = 1024 vs threshold = ∞) used to isolate the
writev path on a single endpoint can be reproduced by editing
`HTTP_WRITEV_THRESHOLD` in `include/php_http_server.h` and rebuilding
`true_async_server.so`.
