# TrueAsync Server — Step 1–3 Results (2026-05-02)

Continuation of `PERFORMANCE_ANALYSIS.md` (2026-05-01). Same bench rig:
docker `benchnet`, `wrk -t4 -c64 -d30s`, single TAS worker, perf record
`-g -F999 --call-graph dwarf` for 20 s during the run.

## Per-symbol perf delta — step 1 vs step 3.3

Same bench, perf record on the live PID. % is self time.

| Symbol | Step 1 | Step 3.3 | Δ |
|---|---:|---:|---:|
| `_emalloc` | 1.42% | **1.20%** | −0.22% |
| `_efree` | 0.79% | **0.68%** | −0.11% |
| **Σ allocator** | **2.21%** | **1.88%** | **−0.33%** |
| `uv_timer_start` | 0.51% | (out of top, < 0.30%) | gone |
| `uv_timer_stop` | 1.04% | (out of top, < 0.30%) | gone |
| `fiber_entry` | 0.99% | 0.90% | −0.09% |
| `async_coroutine_execute` | 0.51% | 0.45% | −0.06% |
| `__vdso_clock_gettime` | 1.27% | 1.46% | +0.19% |

### Reading the deltas

**Allocator −0.33% is the slab arena.** That is ~1.5 ms/s of CPU saved at
65k rps, equivalent to roughly 700–1000 fewer per-second alloc/free calls
that used to flow through Zend MM small-bin paths. The bench keeps each
TCP conn alive across many keep-alive requests, so per-conn alloc cost
amortizes — on a workload with short conn lifetimes (no keep-alive) the
slab win will be larger.

**`uv_timer_start/stop` falling off the profile** is the combined effect
of step 1 (fire-and-forget bypasses `http_write_timer_arm/stop` on the
plaintext hot path) and step 3.3 (write_timer arm site is no longer
needed on the steady-state plaintext send_str_owned path; the global
periodic watchdog handles deadline enforcement).

**`__vdso_clock_gettime` +0.19%** is the cost of the two `zend_hrtime()`
calls in step 3.3's deadline-set sites (`http_connection_spawn` for the
read deadline, dispose-end for the keepalive deadline). Each call is a
~25 ns vDSO syscall — 2× per request at 65k rps lines up with the
observed delta. Could be folded into a single per-request timestamp
read if it ever matters.

**Coroutine machinery (`fiber_entry`, `async_coroutine_execute`)** drifts
down ~0.15% combined — likely a side-effect of better cache locality on
the conn slab and the eliminated heap operations on libuv's timer min-
heap, but the delta is small enough to be partly noise.

## What landed

Branch `feat/multishot-alloc-cb`, three repos:

- `php-async` (`ext/async`) — `libuv_io_write` accepts a `free_cb` for
  fire-and-forget writes (`24bb0c1`).
- `php-src` Zend — extends `zend_async_io_write_t` typedef + new
  `ZEND_ASYNC_IO_WRITE_EX` macro (`124622ca59`).
- `true-async-server2`:
  - `c5fd2c9` step 1: fire-and-forget `http_connection_send_str_owned`,
    drops the dead while-loop in `http_connection_send_raw`, switches the
    HTTP/1 dispose hot path to ownership-transfer.
  - `dc548c0` step 3.0: split `http_server_object` into a small
    `http_server_php` PHP wrapper plus a refcounted C-state. Conns hold
    a ref so the C-state outlives the PHP wrapper. Removes the
    walk-and-NULL pattern and unblocks the slab.
  - `29fd57e` step 3.1: chunked slab arena for `http_connection_t` with
    embedded freelist (`next_conn`) and doubly-linked alive list
    (`next_conn`/`prev_conn`). Replaces per-conn ecalloc/efree and the
    O(N) `unregister_connection` walk.
  - `4a14ec8` step 3.3: per-worker periodic `deadline_tick` watchdog
    that walks the alive list every `max(250 ms, min(read, write,
    keepalive)/2)` and force-closes expired conns. Replaces per-conn
    `write_timer` and per-await read-timeout creation. Also enforces
    `keepalive_timeout` for the first time (the field used to be stored
    but never acted on).

## Reproducing

```bash
docker run -d --rm --name tas-bench --network benchnet \
    --cap-add SYS_PTRACE --cap-add SYS_ADMIN --cap-add PERFMON \
    -e WORKERS=1 tas-step33 \
    php -d extension=true_async_server.so /app/minimal-server.php

docker run --rm --network benchnet williamyeh/wrk \
    -t4 -c64 -d30s --latency http://tas-bench:8080/

# perf inside the container, 20s window during wrk
docker exec tas-bench /usr/lib/linux-tools-6.8.0-111/perf record \
    -g -F 999 --call-graph dwarf -p 1 -o /tmp/perf.data -- sleep 22
docker exec tas-bench /usr/lib/linux-tools-6.8.0-111/perf report \
    -i /tmp/perf.data --stdio --no-children --sort overhead,symbol \
    --no-call-graph
```

`tas-step33` is built locally from `feat/multishot-alloc-cb` HEAD by
copying `sapi/cli/php` and `modules/true_async_server.so` over the
published `tas-final` base image.

## Where the remaining time goes

The post-step-3 profile no longer shows a single dominant overhead the
way the baseline did. The largest individual symbols are now:

- `llhttp__internal__run` 2.35% — HTTP/1 parser; could go lower with a
  hand-tuned llhttp build but mostly compute-bound.
- `_emalloc` 1.20% + `_efree` 0.68% — most remaining alloc is per-
  request (`http1_request_ctx_t`, HttpRequest/HttpResponse zvals).
  Step 5 in PLAN.md targets these.
- `__vdso_clock_gettime` 1.46% — multiple call sites; most are `zend_hrtime`
  in our code (telemetry sojourn/service samples + deadline-set).
  Could be coalesced.
- `execute_ex` 1.09% — Zend VM running the user closure.

Next mechanical gains by PLAN.md priority: vector-write (step 4,
removes one alloc + memcpy per response), coroutine context pool
(step 5, ~1–2% of the remaining `_emalloc`).
