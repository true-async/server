# Step 4.1 — gated per-request hrtime stamps

**Branch**: `feat/multishot-alloc-cb` (`f4c66b8`)
**Date**: 2026-05-02
**Bench**: `wrk -t4 -c64 -d10s` over docker `benchnet`, 1 worker.

## What changed

`http_server_object` now carries a single `bool sample_stamps_enabled`,
computed once at `start()`:

```c
sample_stamps_enabled = (codel_target_ns != 0) || telemetry_enabled;
```

Drain (`max_connection_age_ns`) does **not** require stamps because it
already falls back to a fresh `zend_hrtime()` when `req->end_ns == 0`.

The H1/H2/H3 hot paths gate three sites on this flag:

| site | file:line |
|---|---|
| `req->enqueue_ns = zend_hrtime()` | `http_connection.c:111`, `http2_strategy.c:155`, `http3_dispatch.c:135` |
| `req->start_ns   = zend_hrtime()` | `http_connection.c:1490`, `http2_strategy.c:204`, `http3_dispatch.c:148` |
| `req->end_ns + on_request_sample` | `http_connection.c:1510`, `http2_strategy.c:218`, `http3_dispatch.c:175` |

`total_requests++` was split out of `on_request_sample` into a new
`http_server_count_request()` so request counting keeps working when
the sample is fully gated off.

## RPS — four-corner matrix

5 wrk runs per variant, sorted, median taken (single-thread bench in
WSL2/docker is noisy — see PLAN.md §"Бенчмарк-методология"; medians are
the only honest summary).

| variant | CoDel | Telemetry | stamps | rps median | p99 median |
|---|---|---|:---:|---:|---:|
| **A** (default) | on (5 ms) | off | **on**  | 53 263 | 2.77 ms |
| **B** (no consumer) | off | off | **off** | **55 921** | 2.81 ms |
| C | on (5 ms) | on  | on  | 51 999 | 2.77 ms |
| D | off | on  | on  | 54 556 | 2.78 ms |

Run-to-run noise (~50k–66k swing per variant) dwarfs the inter-variant
gap, so individual rps numbers are not trustworthy in isolation.

The **only consistent signal** in rps is **B above the rest** — every
variant where stamps stay enabled (A, C, D) shows a similar median
~52–55k while B (gated path) sits a bit higher. p99 is identical across
all four within ~50 µs.

## Perf — direct evidence the gate fires

Same wrk session, `perf record -F 999 -g --call-graph dwarf` for 18 s.
Filter: hrtime / sample-call symbols.

### A — CoDel on, telemetry off (stamps **ON**)

```
0.81%  __vdso_clock_gettime
0.41%  http_handler_coroutine_entry
0.27%  http_connection_on_request_ready
0.13%  http_server_on_request_sample
0.05%  http_server_count_request
0.02%  http_server_count_request@plt
0.01%  http_server_on_request_sample@plt
```

### B — CoDel off, telemetry off (stamps **OFF**, fast path)

```
0.65%  __vdso_clock_gettime
0.36%  http_handler_coroutine_entry
0.31%  http_connection_on_request_ready
0.07%  http_server_count_request
0.01%  http_server_count_request@plt
       — http_server_on_request_sample no longer in top symbols —
```

What this proves:

1. **`__vdso_clock_gettime` drops 0.81 → 0.65% (−20% relative).** That
   matches "3 fewer `zend_hrtime` per request": A still has the deadline
   watchdog tick + a couple of incidental hrtime sites; B keeps those
   but loses the per-request trio.
2. **`http_server_on_request_sample` is gone from B's profile.** The
   gate (`if (req && stamps)`) is fully eliminating the call when
   `sample_stamps_enabled == false`.
3. **`http_server_count_request` stays at ~0.07% in both.** Request
   counting still works — that was the explicit goal of the split.
4. **`http_handler_coroutine_entry` drops 0.41 → 0.36%.** It was
   responsible for two of the three stamps and the sample call; less
   work inside it under the gate is consistent.

The vdso saving in absolute terms: at ~55k rps × 0.16 percentage points
out of ~5.5 s of CPU/sec ≈ **8.8 ms/sec saved on hrtime alone**. Three
hrtimes/req at ~25 ns each = 75 ns × 55k = ~4.1 ms/sec; the remainder
is removed call-frame setup for `on_request_sample` and friends.

## Conclusion

- The gate works exactly as designed: when CoDel and telemetry are both
  off, all three per-request stamps and `on_request_sample` disappear
  from the hot path. `total_requests` keeps counting.
- vdso/hrtime cost on the hot path is measurably lower (−20% relative
  to the previous default).
- rps changes are within run-to-run noise on this WSL2/docker setup;
  perf is the better signal for confirming the optimization landed.
- No semantic change for users who turn CoDel or telemetry on — the
  flag flips automatically, stamps come back, behaviour is identical to
  before.

## Reproducing

```bash
# Build image with current php + true_async_server.so on top of tas-final.
mkdir -p /tmp/tas-step41/tas
cp /home/edmond/php-src/sapi/cli/php /tmp/tas-step41/tas/php
cp modules/true_async_server.so      /tmp/tas-step41/tas/

cat > /tmp/tas-step41/Dockerfile <<'EOF'
FROM tas-final:latest
COPY tas/php /usr/local/bin/php
COPY tas/true_async_server.so /usr/local/lib/php/extensions/no-debug-zts-20250926/true_async_server.so
COPY tas/true_async_server.so /usr/local/lib/php/extensions/no-debug-zts-20250000/true_async_server.so
COPY tas/server-variant.php /app/server-variant.php
EOF

docker build -t tas-step41 /tmp/tas-step41

# Variant runner: CODEL_MS=0/5, TELEMETRY=0/1
docker run -d --rm --name tas-bench --network benchnet \
    -e CODEL_MS=0 -e TELEMETRY=0 tas-step41 \
    php -d extension=true_async_server.so /app/server-variant.php

docker run --rm --network benchnet williamyeh/wrk \
    -t4 -c64 -d10s --latency http://tas-bench:8080/
```

`server-variant.php` is the minimal-server with two `getenv()`-driven
setters added (`setBackpressureTargetMs`, `setTelemetryEnabled`).
